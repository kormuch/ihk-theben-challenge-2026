# ADR-2026-07-03-004 - EU DPP MVP module in product-layer

## Context
`Thebenpaul/goal_dpp.md` requires a Digital Product Passport module for the product layer. The module must expose domain-owned product DPP records, JRC three-tier field metadata, role-based views, public no-login access through a stable URL/Data Matrix carrier, lifecycle/version metadata, and audit logging. `Thebenpaul/goal.md` keeps the lakehouse governance model as the architecture source of truth.

## Decision
Implement the DPP MVP inside `ihk-theben-challenge-2026/product-layer` as a curated product-domain projection over the existing product data product.

The canonical DPP record is generated by `product-layer/app/app.py` and includes:
- EU DPP record metadata and standards alignment.
- JRC Essential, Strongly Recommended, and Voluntary field classifications.
- Model, batch, and item granularity.
- Consumer, B2B, and authority access views.
- GTIN, batch/lot, serial number, GS1 Data Matrix payload, and stable public DPP URL.
- Lifecycle, version history, quality status, and lightweight audit events.

## Alternatives considered
- Keep the earlier section-level DPP object model only.
- Split DPP into a separate service before the product-layer MVP is stable.
- Require the data-layer to serve DPP records directly.

## Rationale
The product layer already owns curated product meaning, quality rules, identity, metadata, and API/UI surfaces. Keeping the DPP module in this domain layer matches the lakehouse module approach: data-layer interfaces stay upstream, while product-layer publishes governed curated DPP outputs.

## Impact
Product-layer now owns the DPP API and public HTML surface for the MVP. Data-layer remains an upstream source interface and lakehouse module dependency. Governance is expressed through metadata, access-control configuration, row-level security, masking, certification metadata, and explicit production TODOs for IAM, TLS, catalog publishing, monitoring, and Data Matrix symbol verification.
